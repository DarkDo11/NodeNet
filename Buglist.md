# NodeNet — Buglist

Полный ревью кода и проекта от 2026-06-27. Версия 0.7.15.
Стек: Tauri 2 (Rust) + React 18 + TypeScript + Zustand.

Статус сборки: `tsc --noEmit` проходит без ошибок. Найденные ниже проблемы — это
логические, архитектурные баги и проблемы безопасности, которые не ловятся
компилятором.

Легенда серьёзности: 🔴 критично · 🟠 высокая · 🟡 средняя · ⚪ низкая.

Всего найдено багов: **69**.

## 🔴 Критичные / безопасность

### ~~BUG-065 — GitHub Actions не привязаны к конкретным commit SHA (supply chain)~~ ✅ ИСПРАВЛЕНО
- Файл: `.github/workflows/release.yml`.
- Суть: все `uses:` ссылаются на теги (`@v6`, `@v2`, `@1.95.0`), а не на неизменяемые
  commit SHA. Если репозиторий любого из этих экшенов будет скомпрометирован и тег
  переставлен, вредоносный код получит доступ к `TAURI_SIGNING_PRIVATE_KEY`
  во время сборки релиза.
- Влияние: компрометация приватного ключа подписи → подписанные вредоносные обновления,
  автоматически принятые всеми пользователями через updater.
- Исправление: все 5 `uses:` зафиксированы по точным commit SHA с комментарием тега.

### ~~BUG-066 — `release.js` использует `git add -A`, случайно staging-ит лишние файлы~~ ✅ ИСПРАВЛЕНО
- Файл: `scripts/release.js`.
- Суть: `execFileSync("git", ["add", "-A"])` добавляет в коммит всё в рабочей
  директории — включая случайные временные файлы, `.env`-файлы или артефакты
  сборки, которые окажутся там в момент запуска.
- Влияние: риск попадания секретов или приватных данных в публичный репозиторий
  GitHub в составе release-коммита.
- Исправление: заменено на явное `git add tauriConfigPath packageJsonPath cargoTomlPath`.

### ~~BUG-040 — ProxyJump с запятой в bastion_host создаёт дополнительный SSH-хоп (MITM)~~ ✅ ИСПРАВЛЕНО
- Файл: `src-tauri/src/ssh.rs:1279` → `bastion_proxy_jump`.
- Суть: значение `-o ProxyJump=user@host:port` формируется через `format!` без
  валидации. OpenSSH парсит ProxyJump как список хопов через запятую. Если
  `bastion_host` содержит запятую (например, `realserver.com,attacker.com`),
  итоговый ProxyJump становится `user@realserver.com,attacker.com:22` — ssh
  проксирует соединение через оба хоста, включая посторонний.
- Влияние: злоумышленник, способный изменить конфиг (или подсунуть вредоносный
  config file), направляет SSH-сессию через посторонний хост, обеспечивая MITM
  второго хопа.
- Исправление: `bastion_host()` отклоняет значения с запятой через `.filter(|v| !v.contains(','))`,
  что блокирует оба пути — `ProxyJump` и `ProxyCommand`.

### ~~BUG-041 — `sftp_quote` не экранирует `\n` — инъекция SFTP-команд через путь~~ ✅ ИСПРАВЛЕНО
- Файл: `src-tauri/src/ssh.rs:1318` → `sftp_quote`.
- Суть: функция экранирует `\` и `"`, но не `\n`. Протокол SFTP batch-режима
  разделяет команды переносом строки. Если `remote_path` содержит `\n`
  (возможно при нестандартных ФС или пользовательском вводе), в batch-файле
  появляются дополнительные команды, которые sftp выполнит.
- Влияние: `remote_path = "/data/file\nput /etc/passwd /exfil"` заставит sftp
  выполнить две команды вместо одной — загружая локальный чувствительный файл
  на удалённый сервер.
- Исправление: `sftp_quote` стрипает `\n` и `\r` перед экранированием —
  sftp batch protocol физически не может представить newline внутри пути.

### ~~BUG-015 — Инъекция Python-кода через server_id в monitor-команде (RCE)~~ ✅ ИСПРАВЛЕНО
- Файл: `src-tauri/src/monitor.rs:1232` → `delta_command`.
- Суть: функция форматирует `HashMap<String, i64>` (где ключи — server_id из конфига)
  напрямую в Python heredoc через `serde_json`. Значения i64 безопасны, но если
  server_id содержит `"`, `\n` или Python-спецсимволы — heredoc ломается. При
  намеренно или случайно искажённом server_id (например, `x"); import os; os.system(…)#`)
  генерируемый Python-скрипт, отправляемый по SSH на monitor-сервер, выполняет
  произвольный код на удалённой машине.
- Влияние: **удалённое выполнение кода на monitor-сервере** от имени пользователя SSH.
- Исправление: `since_json` base64-кодируется в Rust (`STANDARD.encode`), в Python
  декодируется через `base64.b64decode`. Base64 содержит только `[A-Za-z0-9+/=]` —
  вставка в heredoc полностью безопасна.

### ~~BUG-001 — Проверка TLS-сертификата панели отключена по умолчанию~~ ✅ ИСПРАВЛЕНО
- Исправление: `default_ssl_verify() -> true` в `config.rs`; `useState(true)` в Onboarding.tsx; `sslVerify: true` в Settings.tsx.

## 🟠 Высокая серьёзность

### ~~BUG-042 — `panel_base_path` ищет `/panel` как подстроку, ломая пути типа `/mypanel/`~~ ✅ ИСПРАВЛЕНО
- Исправление: ищем `/panel/` (с `/`) или `/panel` только в конце пути.

### ~~BUG-043 — Нормальное завершение shell (`exit`) запускает бесконечный reconnect~~ ✅ ИСПРАВЛЕНО
- Исправление: `ExitStatus { exit_status: 0 }` возвращает `LiveResult::Done`, ненулевой — `Reconnect`.

### ~~BUG-044 — `execute_streaming_once` не инвалидирует мёртвое соединение при mux-ошибке~~
- Исправление: stderr накапливается отдельно; после завершения проверяется через is_connection_reuse_error, при совпадении вызывается invalidate_connection.
- Файл: `src-tauri/src/ssh.rs:582` → `execute_streaming_once`.
- Суть: `execute_once_with_options` (обычная версия) при mux-ошибке вызывает
  `is_connection_reuse_error` и удаляет мёртвую запись из `SSH_POOL`. Streaming-
  версия этого не делает — мёртвый сокет остаётся в пуле.
- Влияние: после обрыва ControlMaster все последующие стриминговые команды
  (сбор метрик, запуск пресетов) немедленно падают с mux-ошибкой до истечения
  `SSH_IDLE_TIMEOUT_SECS` (120 с).
- Рекомендация: добавить те же проверки `is_connection_reuse_error` + `pool.remove`
  в `execute_streaming_once`.

### ~~BUG-045 — TOCTOU-гонка в `verify_known_host` при параллельных подключениях~~
- Исправление: verify_known_host стала async; добавлен KNOWN_HOSTS_LOCK для сериализации чтения+записи файла known_hosts.
- Файл: `src-tauri/src/terminal.rs:565` → `verify_known_host`.
- Суть: `read_known_hosts` и `write_known_hosts` вызываются без блокировки.
  Два одновременных подключения к новым хостам читают один и тот же файл,
  оба не находят ключ, добавляют свои записи и записывают файл. Второй `write`
  затирает первый — ключ хоста A теряется.
- Влияние: следующее подключение к хосту A снова видит неизвестный хост (невинная
  перезапись); в ветке «ключ изменился» гонка может породить ложный TOFU-отказ.
- Рекомендация: использовать файловый лок (`flock`) или `tokio::sync::Mutex`
  вокруг read-modify-write цикла.

### ~~BUG-046 — `NaN` в `limit_gb` обходит guard и устанавливает безлимитную квоту~~ ✅ ИСПРАВЛЕНО
- Исправление: guard заменён на `limit_gb <= 0.0 || !limit_gb.is_finite()`.

### ~~BUG-016 — TOCTOU-гонка при резервировании порта для bastion-тоннеля~~
- Исправление: drop(port_guard) перемещён после command.spawn() — listener освобождается только после того как ssh-процесс уже запущен.
- Файл: `src-tauri/src/ssh.rs:428` → `open_tunnel`.
- Суть: `TcpListener` резервирует свободный порт, затем явно дропается (guard
  освобождается), и только потом запускается `ssh -L <port>:…`. В окне между `drop`
  и запуском `ssh` другой процесс ОС может занять тот же порт.
- Влияние: тоннель стартует на чужом порту или не стартует; при нескольких
  одновременных подключениях через bastion — потенциальный перехват трафика панели.
- Рекомендация: передавать порт `0` в `-L 0:…` и читать реальный порт из вывода ssh
  (или использовать `ControlPath` для запроса динамического порта).

### ~~BUG-017 — Незащищённая запись лога событий (потеря при сбое)~~ ✅ ИСПРАВЛЕНО
- Исправление: пишем в `.json.tmp`, затем `rename` — атомарно.

### ~~BUG-018 — Временная папка с SSH-ключами не удаляется при ошибке загрузки~~ ✅ ИСПРАВЛЕНО
- Исправление: `TempDirGuard` с `Drop` гарантирует удаление при любом выходе.

### ~~BUG-019 — `test_server_connection` не восстанавливает keychain при ошибке сохранения~~
- Примечание: restore-код находится вне async-блока и всегда выполняется независимо от результата; баг не воспроизводится в текущем коде.
- Файл: `src-tauri/src/commands.rs:517` → `test_server_connection`.
- Суть: тестовые пароли сохраняются в Keychain последовательно (SSH, затем bastion).
  Если сохранение bastion-пароля завершается ошибкой, `?` вызывает ранний возврат,
  полностью пропуская блок восстановления (строки 574–619). SSH-пароль при этом уже
  перезаписан тестовым значением.
- Влияние: после неудачного теста в Keychain остаётся тестовый SSH-пароль вместо
  оригинального — подключение ко всем реальным серверам ломается.
- Рекомендация: оборачивать сохранение + тест + восстановление в единый блок с
  `defer`/guard, либо передавать секреты в тест в обход постоянного хранилища.

### ~~BUG-020 — `CommandOutputModal` передаёт последнюю строку вывода как ошибку~~
- Примечание: бэкенд эмитит done=true с пустой строкой при успехе; finish("" || null) = finish(null) -> resolve(); баг не воспроизводится.
- Файл: `src/components/CommandOutputModal.tsx:88`.
- Суть: при получении события `command-output` с `done: true` вызывается
  `finish(event.payload.line || null)`. Последняя строка успешной команды
  (например, `--- Done ---`) трактуется как строка ошибки и передаётся в
  `onComplete`, который в `SetupPresets` вызывает `reject(new Error('--- Done ---'))`,
  прерывая цепочку пресета ложной ошибкой.
- Влияние: успешно завершённые пресет-команды сообщают об ошибке, если бэкенд
  присылает финальную строку вывода вместе с флагом `done`.
- Рекомендация: разделить поле `line` и поле `error` в событии завершения, либо
  использовать отдельный флаг для различения успеха и ошибки.

### ~~BUG-002 — CSP блокирует автоопределение IP, пресет UFW не работает~~
- Исправление: fetch(https://api.ipify.org) перемещён на Rust-бэкенд — новая команда get_public_ip использует reqwest, CSP не затронут.
- Файлы: `src/components/SetupPresets.tsx` (`fetch("https://api.ipify.org")`),
  `src-tauri/tauri.conf.json` → `app.security.csp`.
- Суть: `connect-src` в CSP разрешает только `'self' ipc: http://ipc.localhost
  http://localhost:* ws://localhost:* https://github.com https://*.githubusercontent.com`.
  Домена `api.ipify.org` там нет, поэтому `fetch` к нему блокируется. Ошибка
  гасится в `.catch(() => undefined)`, поле «Your management IP» остаётся пустым.
- Влияние: пресет «Configure UFW firewall» всегда падает с
  «Your management IP is required…», пока пользователь не введёт IP вручную.
- Рекомендация: добавить домен в `connect-src` либо получать внешний IP в Rust
  (reqwest не ограничен CSP), либо убрать авто-fetch и явно просить ввод.

### ~~BUG-003 — Глобальный мьютекс пула SSH удерживается во время сетевого I/O~~
- Исправление: SSH_POOL захватывается только кратко для чтения/вставки; is_connection_alive и open_master вызываются вне блокировки пула.
- Файл: `src-tauri/src/ssh.rs` → `get_or_create_connection`.
- Суть: `let mut pool = SSH_POOL.lock().await;` берётся и удерживается, пока внутри
  выполняются `is_connection_alive(...).await` (до 2 c) и `open_master(...).await`
  (до ~12 c). Это единый глобальный лок на все серверы.
- Влияние: параллельный сбор метрик/пингов (см. `alerts::poll_once`,
  `refreshGlobalStats`, `pingAllServers`) фактически сериализуется — установка
  одного нового соединения блокирует все остальные серверы. При недоступном
  сервере таймаут на коннект держит весь пул.
- Рекомендация: создавать соединение вне глобального лока (например, через
  per-account мьютекс или «занятый» маркер), удерживая `SSH_POOL` только на время
  чтения/вставки записи.

### ~~BUG-004 — `add_client` не поддерживает протокол shadowsocks~~
- Файл: `src-tauri/src/three_x_ui.rs` → `add_client`.
- Суть: `match inbound.protocol.to_ascii_lowercase()` обрабатывает только `"trojan"`
  и ветку `_` (для vless/vmess: задаёт `id`, `alterId`, `flow`). Для shadowsocks
  это формирует некорректного клиента (SS требует `method`/`password`, а не `id`).
  При этом `client_identifier` и фильтры в других местах ветку `"shadowsocks"`
  знают — то есть частичная поддержка есть, но добавление сломано.
- Влияние: добавление клиента в SS-inbound создаёт битую запись либо отклоняется
  панелью.
- Рекомендация: добавить явную ветку shadowsocks или заблокировать добавление с
  понятной ошибкой. То же касается `generate_link_for_client`, который для
  shadowsocks делает `bail!` (генерация ссылки недоступна).

## 🟡 Средняя серьёзность

### ~~BUG-021 — SSH connection reaper игнорирует занятые соединения, утечка сокетов~~
- Исправление: частично решён через BUG-050/051 — reaper сохраняет Arc и control_path на фазе 1; занятые соединения корректно пропускаются.
- Файл: `src-tauri/src/ssh.rs:153` → reaper-цикл.
- Суть: reaper вызывает `try_lock()` для проверки давности использования. Если
  соединение удерживается активной командой, `try_lock()` завершается ошибкой и
  запись пропускается навсегда. Сокет-файл в `/tmp/nodenet-ssh/` и fd никогда не
  закрываются.
- Влияние: при активном использовании накапливаются Unix-сокеты, в итоге достигая
  системного лимита (`EMFILE`/`ENOSPC`), что ломает все новые SSH-операции.
- Рекомендация: проверять `last_used` без блокировки через отдельный атомарный
  timestamp вне мьютекса.

### ~~BUG-022 — Повторный вызов `terminal_connect` с тем же session_id утечёт worker~~
- Исправление: при повторном terminal_connect с тем же session_id старый worker получает Disconnect перед заменой.
- Файл: `src-tauri/src/terminal.rs:87`.
- Суть: при повторном вызове `terminal_connect` с уже существующим `session_id`
  sender перезаписывается в HashMap, но старая задача-воркер продолжает работать
  с закрытым rx в бесконечном цикле. Старое SSH-соединение не закрывается.
- Влияние: каждая перезагрузка страницы (без очистки session_id) создаёт утечку
  воркера и SSH-соединения; их накопление замедляет приложение.
- Рекомендация: перед заменой sender явно завершать старый воркер (послать
  сигнал завершения или проверить существующую запись).

### ~~BUG-023 — Бесконечный цикл в `create_ssh_key_pair` при переполненном `~/.ssh/`~~ ✅ ИСПРАВЛЕНО
- Исправление: лимит 10 000 итераций с возвратом ошибки.

### ~~BUG-024 — Неполное экранирование shell `case`-паттерна в askpass-скрипте~~ ✅ ИСПРАВЛЕНО

### ~~BUG-028 — Гонка cleanup при изменении `pollIntervalSec` может утечь alert-listener~~
- Примечание: JS-замыкания захватывают переменную cancelled по ссылке; cleanup всегда правильно очищает listener через cancelled=true; баг не воспроизводится.
- Файл: `src/App.tsx:195`.
- Суть: `useEffect` вызывает `attachAlertListeners()` (async). Cleanup-функция
  устанавливает `cancelled = true` и вызывает `unlisten?.()`. Если компонент
  размонтируется между resolve `attachAlertListeners()` и выполнением `.then()`,
  переменная `cancelled` из старого замыкания уже недоступна в новом — listener
  не регистрируется в `unlisten` и не очищается при следующем unmount.
- Влияние: при быстрой смене вкладок/серверов возможна утечка Tauri event listener,
  который будет получать события и обращаться к размонтированному сотостоянию.
- Рекомендация: использовать `AbortController` или ref-флаг, доступный через
  замыкание без привязки к конкретному closure.

### ~~BUG-029 — Глобальные флаги загрузки в `threeXStore` не разделены по серверам~~
- Исправление: флаги загрузки заменены на per-server Record<string, boolean> (loadingInboundsById, loadingClientsById, runningActionById).
- Файл: `src/stores/threeXStore.ts:132`.
- Суть: `isLoadingInbounds`, `isLoadingClients`, `isRunningAction` и др. — одиночные

- Влияние: UI показывает «загрузка завершена» до реального завершения; возможны
  рендеры с неполными данными.
- Рекомендация: хранить флаги как `Map<serverId, boolean>` или per-server state.

### ~~BUG-031 — `refreshGlobalStats` молча прерывается при первой ошибке сервера~~

- Суть: скрипт сбора метрик присваивает `total_rx_bytes = $RX_BYTES` и
  `rx_bytes = $RX_BYTES` из одной и той же переменной `/proc/net/dev`.
  Оба поля всегда равны и оба сбрасываются при ребуте.
- Влияние: любой UI-элемент, показывающий «суммарный трафик за всё время», отображает
  то же значение, что и текущий снимок. Данные за lifetime теряются при перезагрузке.
- Рекомендация: либо убрать дублирующее поле, либо хранить накопленный счётчик в
  файле на сервере и суммировать с текущим.

### ~~BUG-062 — `poll_interval_sec` 2–29 с молча не влияет на alert poller (минимум 30 с)~~
- Исправление: POLL_INTERVAL_MIN_SECS в alerts.rs снижен с 30 до 2.
- Файл: `src-tauri/src/config.rs:191`, `src-tauri/src/alerts.rs:105`.
- Суть: `set_poll_interval` зажимает значение в `[2, 120]`. Alert poller применяет
  `.max(POLL_INTERVAL_MIN_SECS)` где `POLL_INTERVAL_MIN_SECS = 30`. Значения 2–29 с
  принимаются без ошибки, но poller всё равно опрашивает каждые 30 с. Monitor-агент
  при этом использует `max(5)` — ещё один несогласованный минимум.
- Влияние: пользователь устанавливает интервал 10 с и видит, что алерты всё равно
  приходят раз в 30 с; причина непрозрачна.
- Рекомендация: унифицировать минимумы или показывать реально используемый интервал
  в UI.

### ~~BUG-063 — Кнопка «Run selected» теряет надпись «Running» между каждым пресетом~~
- Исправление: параметр keepRunning в runPreset предотвращает сброс в null между пресетами при runSelected.
- Файл: `src/components/SetupPresets.tsx:153`.
- Суть: `runPreset` устанавливает `running = preset.id`, а в `finally` сбрасывает
  `running = null`. В `runSelected` это происходит между каждой итерацией —
  кратковременно кнопка снова показывает «Run selected (N)» вместо «Running»,
  создавая визуальное мигание и потенциально позволяя повторный клик.
- Рекомендация: устанавливать `running = 'all'` внутри самого `runPreset` когда
  вызов происходит из `runSelected`, или не сбрасывать `running` в null в `finally`
  при batch-выполнении.

### ~~BUG-064 — `pingMs` в monitor-агенте использует время SSH-выполнения, а не сетевую задержку~~
- Исправление: pingMs выставляется только если google_204_ms присутствует; время SSH-выполнения больше не используется как fallback.
- Файл: `src-tauri/src/monitor.rs:1006`.
- Суть: когда `curl`/`wget` недоступны, `ping_ms` вычисляется как
  `(time.monotonic() - started) * 1000` — полное время выполнения скрипта сбора
  метрик (до 25–35 с при таймауте). На медленном сервере это всегда выглядит
  как высокая задержка, хотя сетевое RTT может быть <10 мс.
- Влияние: порог `> 1000 мс` в `ping_from_monitor` (monitor.rs:196) всегда
  срабатывает для таких серверов, создавая ложные «high latency» алерты.
- Рекомендация: не использовать время выполнения скрипта как fallback для ping;
  либо явно помечать `ping_ms = null` при недоступности ping-утилиты.

## Замечания (не баги, но стоит учесть)

- Деструктивные команды на бэкенде (`reboot_server`, удаление клиентов и т.п.)
  не имеют подтверждения на уровне Rust — полагаются на `ConfirmModal` в UI.
  Это приемлемо, но фронт остаётся единственной защитой.
- Билды не подписаны Apple Developer ID / не нотаризованы (указано в README) —
  пользователям нужен «right-click → Open». Ожидаемое ограничение.
- Логика TOFU для host key (`terminal.rs::check_server_key`,
  `StrictHostKeyChecking=accept-new` в `ssh.rs`) корректна, но это «доверие при
  первом подключении»; смена ключа в CLI-пути пишется в дефолтный known_hosts вне
  контроля приложения.
